46 # Introduction
47 
48 Reflection (or rather, its lack) is a trending topic in the C++ world. I looked
49 for a third-party library that met my needs on the subject, but I always came
50 across some details that I didn't like: macros, being intrusive, too many
51 allocations.<br/>
52 In one word: **unsatisfactory**.
53 
54 I finally decided to write a built-in, non-intrusive and macro-free runtime
55 reflection system for my own.<br/>
56 Maybe I didn't do better than others or maybe yes. Time will tell me.

60 ## Requirements
61 
62 To be able to use `meta`, users must provide a full-featured compiler that
63 supports at least C++17.<br/>
64 The requirements below are mandatory to compile the tests and to extract the
65 documentation:
66 
67 * CMake version 3.2 or later.
68 * Doxygen version 1.8 or later.

70 If you are looking for a C++14 version of `meta`, feel free to
71 [contact me](https://github.com/skypjack).

73 ## Library
74 
75 `meta` is a header-only library. This means that including the `factory.hpp` and
76 `meta.hpp` headers is enough to include the library as a whole and use it.<br/>
77 It's a matter of adding the following lines to the top of a file:
78 
79 ```cpp
80 #include <meta/factory.hpp>
81 #include <meta/meta.hpp>
82 ```
83 
84 Then pass the proper `-I` argument to the compiler to add the `src` directory to
85 the include paths.

124 # Crash course

126 ## Reflection in a nutshell
127 
128 Reflection always starts from real types (users cannot reflect imaginary types
129 and it would not make much sense, we wouldn't be talking about reflection
130 anymore).<br/>
131 To _reflect_ a type, the library provides the `reflect` function:
132 
133 ```cpp
134 meta::factory factory = meta::reflect<my_type>("reflected_type");
135 ```
136 
137 It accepts the type to reflect as a template parameter and an optional name as
138 an argument. Names are important because users can retrieve meta types at
139 runtime by searching for them by name. However, there are cases in which users
140 can be interested in adding features to a reflected type so that the reflection
141 system can use it correctly under the hood, but they don't want to allow
142 searching the type by name.<br/>
143 In both cases, the returned value is a factory object to use to continue
144 building the meta type.
145 
146 A factory is such that all its member functions returns the factory itself.
147 It can be used to extend the reflected type and add the following:
148 
149 * _Constructors_. Actual constructors can be assigned to a reflected type by
150  specifying their list of arguments. Free functions (namely, factories) can be
151  used as well, as long as the return type is the expected one. From a client's
152  point of view, nothing changes if a constructor is a free function or an
153  actual constructor.<br/>
154  Use the `ctor` member function for this purpose:
155 
156  ```cpp
157  meta::reflect<my_type>("reflected").ctor<int, char>().ctor<&factory>();
158  ```
159 
160 * _Destructors_. Free functions can be set as destructors of reflected types.
161  The purpose is to give users the ability to free up resources that require
162  special treatment before an object is actually destroyed.<br/>
163  Use the `dtor` member function for this purpose:
164 
165  ```cpp
166  meta::reflect<my_type>("reflected").dtor<&destroy>();
167  ```
168 
169 * _Data members_. Both real data members of the underlying type and static and
170  global variables, as well as constants of any kind, can be attached to a meta
171  type. From a client's point of view, all the variables associated with the
172  reflected type will appear as if they were part of the type itself.<br/>
173  Use the `data` member function for this purpose:
174 
175  ```cpp
176  meta::reflect<my_type>("reflected")
177  .data<&my_type::static_variable>("static")
178  .data<&my_type::data_member>("member")
179  .data<&global_variable>("global");
180  ```
181 
182  This function requires as an argument the name to give to the meta data once
183  created. Users can then access meta data at runtime by searching for them by
184  name.<br/>
185  Data members can be set also by means of a couple of functions, namely a
186  setter and a getter. Setters and getters can be either free functions, member
187  functions or mixed ones, as long as they respect the required signatures.<br/>
188  Refer to the inline documentation for all the details.
189 
190 * _Member functions_. Both real member functions of the underlying type and free
191  functions can be attached to a meta type. From a client's point of view, all
192  the functions associated with the reflected type will appear as if they were
193  part of the type itself.<br/>
194  Use the `func` member function for this purpose:
195 
196  ```cpp
197  meta::reflect<my_type>("reflected")
198  .func<&my_type::static_function>("static")
199  .func<&my_type::member_function>("member")
200  .func<&free_function>("free");
201  ```
202 
203  This function requires as an argument the name to give to the meta function
204  once created. Users can then access meta functions at runtime by searching for
205  them by name.
206 
207 * _Base classes_. A base class is such that the underlying type is actually
208  derived from it. In this case, the reflection system tracks the relationship
209  and allows for implicit casts at runtime when required.<br/>
210  Use the `base` member function for this purpose:
211 
212  ```cpp
213  meta::reflect<derived_type>("derived").base<base_type>();
214  ```
215 
216  From now on, wherever a `base_type` is required, an instance of `derived_type`
217  will also be accepted.
218 
219 * _Conversion functions_. Actual types can be converted, this is a fact. Just
220  think of the relationship between a `double` and an `int` to see it. Similar
221  to bases, conversion functions allow users to define conversions that will be
222  implicitly performed by the reflection system when required.<br/>
223  Use the `conv` member function for this purpose:
224 
225  ```cpp
226  meta::reflect<double>().conv<int>();
227  ```
228 
229 That's all, everything users need to create meta types and enjoy the reflection
230 system. At first glance it may not seem that much, but users usually learn to
231 appreciate it over time.<br/>
232 Also, do not forget what these few lines hide under the hood: a built-in,
233 non-intrusive and macro-free system for reflection in C++. Features that are
234 definitely worth the price, at least for me.
235 
236 ## Any as in any type
237 
238 The reflection system comes with its own meta any type. It may seem redundant
239 since C++17 introduced `std::any`, but it is not.<br/>
240 In fact, the _type_ returned by an `std::any` is a const reference to an
241 `std::type_info`, an implementation defined class that's not something everyone
242 wants to see in a software. Furthermore, the class `std::type_info` suffers from
243 some design flaws and there is even no way to _convert_ an `std::type_info` into
244 a meta type, thus linking the two worlds.
245 
246 A meta any object provides an API similar to that of its most famous counterpart
247 and serves the same purpose of being an opaque container for any type of
248 value.<br/>
249 It minimizes the allocations required, which are almost absent thanks to _SBO_
250 techniques. In fact, unless users deal with _fat types_ and create instances of
251 them though the reflection system, allocations are at zero.
252 
253 A meta any object can be created by any other object or as an empty container
254 to initialize later:
255 
256 ```cpp
257 // a meta any object that contains an int
258 meta::any any{0};
259 
260 // an empty meta any object
261 meta::any empty{};
262 ```
263 
264 It can be constructed or assigned by copy and move and it takes the burden of
265 destroying the contained object when required.<br/>
266 A meta any object has a `type` member function that returns the meta type of the
267 contained value, if any. The member functions `can_cast` and `can_convert` are
268 used to know if the underlying object has a given type as a base or if it can be
269 converted implicitly to it. Similarly, `cast` and `convert` do what they promise
270 and return the expected value.

272 ## Enjoy the runtime
273 
274 Once the web of reflected types has been constructed, it's a matter of using it
275 at runtime where required.
276 
277 To search for a reflected type there are two options: by type or by name. In
278 both cases, the search can be done by means of the `resolve` function:
279 
280 ```cpp
281 // search for a reflected type by type
282 meta::type by_type = meta::resolve<my_type>();
283 
284 // search for a reflected type by name
285 meta::type by_name = meta::resolve("reflected_type");
286 ```
287 
288 There exits also a third overload of the `resolve` function to use to iterate
289 all the reflected types at once:
290 
291 ```cpp
292 resolve([](meta::type type) {
293  // ...
294 });
295 ```
296 
297 In all cases, the returned value is an instance of `type`. This type of objects
298 offer an API to know the _runtime name_ of the type, to iterate all the meta
299 objects associated with them and even to build or destroy instances of the
300 underlying type.<br/>
301 Refer to the inline documentation for all the details.
302 
303 The meta objects that compose a meta type are accessed in the following ways:
304 
305 * _Meta constructors_. They are accessed by types of arguments:
306 
307  ```cpp
308  meta::ctor ctor = meta::resolve<my_type>().ctor<int, char>();
309  ```
310 
311  The returned type is `ctor` and may be invalid if there is no constructor that
312  accepts the supplied arguments or at least some types from which they are
313  derived or to which they can be converted.<br/>
314  A meta constructor offers an API to know the number of arguments, the expected
315  meta types and to invoke it, therefore to construct a new instance of the
316  underlying type.
317 
318 * _Meta destructor_. It's returned by a dedicated function:
319 
320  ```cpp
321  meta::dtor dtor = meta::resolve<my_type>().dtor();
322  ```
323 
324  The returned type is `dtor` and may be invalid if there is no custom
325  destructor set for the given meta type.<br/>
326  All what a meta destructor has to offer is a way to invoke it on a given
327  instance. Be aware that the result may not be what is expected.
328 
329 * _Meta data_. They are accessed by name:
330 
331  ```cpp
332  meta::data data = meta::resolve<my_type>().data("member");
333  ```
334 
335  The returned type is `data` and may be invalid if there is no meta data object
336  associated with the given name.<br/>
337  A meta data object offers an API to query the underlying type (ie to know if
338  it's a const or a static one), to get the meta type of the variable and to set
339  or get the contained value.
340 
341 * _Meta functions_. They are accessed by name:
342 
343  ```cpp
344  meta::func func = meta::resolve<my_type>().func("member");
345  ```
346 
347  The returned type is `func` and may be invalid if there is no meta function
348  object associated with the given name.<br/>
349  A meta function object offers an API to query the underlying type (ie to know
350  if it's a const or a static function), to know the number of arguments, the
351  meta return type and the meta types of the parameters. In addition, a meta
352  function object can be used to invoke the underlying function and then get the
353  return value in the form of meta any object.
354 
355 
356 * _Meta bases_. They are accessed through the name of the base types:
357 
358  ```cpp
359  meta::base base = meta::resolve<derived_type>().base("base");
360  ```
361 
362  The returned type is `base` and may be invalid if there is no meta base object
363  associated with the given name.<br/>
364  Meta bases aren't meant to be used directly, even though they are freely
365  accessible. They expose only a few methods to use to know the meta type of the
366  base class and to convert a raw pointer between types.
367 
368 * _Meta conversion functions_. They are accessed by type:
369 
370  ```cpp
371  meta::conv conv = meta::resolve<double>().conv<int>();
372  ```
373 
374  The returned type is `conv` and may be invalid if there is no meta conversion
375  function associated with the given type.<br/>
376  The meta conversion functions are as thin as the meta bases and with a very
377  similar interface. The sole difference is that they return a newly created
378  instance wrapped in a meta any object when they convert between different
379  types.
380 
381 All the objects thus obtained as well as the meta types can be explicitly
382 converted to a boolean value to check if they are valid:
383 
384 ```cpp
385 meta::func func = meta::resolve<my_type>().func("member");
386 
387 if(func) {
388  // ...
389 }
390 ```
391 
392 Furthermore, all meta objects with the exception of meta destructors can be
393 iterated through an overload that accepts a callback through which to return
394 them. As an example:
395 
396 ```cpp
397 meta::resolve<my_type>().data([](meta::data data) {
398  // ...
399 });
400 ```
401 
402 A meta type can also be used to `construct` or `destroy` actual instances of the
403 underlying type.<br/>
404 In particular, the `construct` member function accepts a variable number of
405 arguments and searches for a match. It returns a `any` object that may or may
406 not be initialized, depending on whether a suitable constructor has been found
407 or not. On the other side, the `destroy` member function accepts instances of
408 `any` as well as actual objects by reference and invokes the registered
409 destructor if any or a default one.<br/>
410 Be aware that the result of a call to `destroy` may not be what is expected.
411 
412 Meta types and meta objects in general contain much more than what is said: a
413 plethora of functions in addition to those listed whose purposes and uses go
414 unfortunately beyond the scope of this document.<br/>
415 I invite anyone interested in the subject to look at the code, experiment and
416 read the official documentation to get the best out of this powerful tool.

418 ## Named constants and enums
419 
420 A special mention should be made for constant values and enums. It wouldn't be
421 necessary, but it will help distracted readers.
422 
423 As mentioned, the `data` member function can be used to reflect constants of any
424 type among the other things.<br/>
425 This allows users to create meta types for enums that will work exactly like any
426 other meta type built from a class. Similarly, arithmetic types can be enriched
427 with constants of special meaning where required.<br/>
428 Personally, I find it very useful not to export what is the difference between
429 enums and classes in C++ directly in the space of the reflected types.
430 
431 All the values thus exported will appear to users as if they were constant data
432 members of the reflected types.
433 
434 Exporting constant values or elements from an enum is as simple as ever:
435 
436 ```cpp
437 meta::reflect<my_enum>()
438  .data<my_enum::a_value>("a_value")
439  .data<my_enum::another_value>("another_value");
440 
441 meta::reflect<int>().data<2048>("max_int");
442 ```
443 
444 It goes without saying that accessing them is trivial as well. It's a matter of
445 doing the following, as with any other data member of a meta type:
446 
447 ```cpp
448 my_enum value = meta::resolve<my_enum>().data("a_value").get({}).cast<my_enum>();
449 int max = meta::resolve<int>().data("max_int").get({}).cast<int>();
450 ```
451 
452 As a side note, remember that all this happens behind the scenes without any
453 allocation because of the small object optimization performed by the meta any
454 class.

456 ## Properties and meta objects
457 
458 Sometimes (ie when it comes to creating an editor) it might be useful to be able
459 to attach properties to the meta objects created. Fortunately, this is possible
460 for most of them.<br/>
461 To attach a property to a meta object, no matter what as long as it supports
462 properties, it is sufficient to provide an object at the time of construction
463 such that `std::get<0>` and `std::get<1>` are valid for it. In other terms, the
464 properties are nothing more than key/value pairs users can put in an
465 `std::pair`. As an example:
466 
467 ```cpp
468 meta::reflect<my_type>("reflected", std::make_pair("tooltip"_hs, "message"));
469 ```
470 
471 The meta objects that support properties offer then a couple of member functions
472 named `prop` to iterate them at once and to search a specific property by key:
473 
474 ```cpp
475 // iterate all the properties of a meta type
476 meta::resolve<my_type>().prop([](meta::prop prop) {
477  // ...
478 });
479 
480 // search for a given property by name
481 meta::prop prop = meta::resolve<my_type>().prop("tooltip"_hs);
482 ```
483 
484 Meta properties are objects having a fairly poor interface, all in all. They
485 only provide the `key` and the `value` member functions to be used to retrieve
486 the key and the value contained in the form of meta any objects, respectively.
487 
488 ## Unregister types
489 
490 A type registered with the reflection system can also be unregistered. This
491 means unregistering all its data members, member functions, conversion functions
492 and so on. However, the base classes won't be unregistered, since they don't
493 necessarily depend on it. Similarly, implicitly generated types (as an example,
494 the meta types implicitly generated for function parameters when needed) won't
495 be unregistered.
496 
497 To unregister a type, users can use the `unregister` function from the global
498 namespace:
499 
500 ```cpp
501 meta::unregister<my_type>();
502 ```
503 
504 This function returns a boolean value that is true if the type is actually
505 registered with the reflection system, false otherwise.<br/>
506 The type can be re-registered later with a completely different name and form.
